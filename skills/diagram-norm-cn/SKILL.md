# 中文论文图件规范

## 总原则

1. 必须先确认图件类型，再画图。不要把功能模块图画成流程图，不要把实体-联系图画成表结构图，不要把用例图画成业务流程图。
2. 必须先联网检索该图类的标准画法或权威说明，再生成图件。除非用户明确禁止联网，否则不能只凭记忆输出。
3. 优先使用标准、官方文档、教材式资料或工具官方文档作为依据。检索后在回复中简要说明参考了哪些规范来源。
4. 默认输出 draw.io/diagrams.net 可导入的 XML 代码。用户只要图片时，也优先保留可编辑源代码。
5. 默认所有图中文字使用中文。除非用户明确要求英文，实体名、属性名、关系名、流程节点、注释、图题都要写中文；数据库字段名要翻译成中文含义。
6. 默认黑白图：黑色线条，白色或透明填充，不使用彩色装饰，不使用阴影，不显示网格。
7. 默认字体使用 `Microsoft YaHei,SimSun`，避免中文乱码。字号要适合论文截图，节点中文字不能拥挤。
8. 图题放在图下方，按论文当前章节编号，例如“图4.8 系统核心业务实体-E-R关联图”。如果用户给定图号，以用户给定为准。

## 强制联网检索流程

每次绘图前执行：

1. 识别图类：实体-联系图、用例图、流程图、活动图、时序图、系统架构图、功能模块图、数据流图等。
2. 用中文和英文关键词各检索一次，例如：
   - “实体 联系 图 关系 菱形 属性 椭圆 标准画法”
   - “用例图 actor system boundary include extend UML standard”
   - “流程图 开始 结束 判断 处理 输入输出 符号 标准”
   - “活动图 泳道 开始 结束 判断 合并 UML 标准”
3. 优先读取以下来源：
   - 标准组织或官方规范页面。
   - diagrams.net/draw.io 官方文档。
   - 高校教材、课程讲义、权威教程。
   - 论文模板或学校格式要求。
4. 不要依赖广告站、低质量博客或无法判断来源的内容。如果只能找到二手资料，要在回复中说明“依据有限”。
5. 检索完成后再画图，并把查到的规范转化为具体图形符号。

## 输出格式

回复结构默认如下：

1. 简短说明图件类型和采用规范。
2. 提供完整 XML 代码块。
3. 如有必要，说明用户如何导入 diagrams.net。

XML 代码块使用：

```xml
...
```

不要在 XML 之外夹杂大量解释。不要输出无法复制导入的残缺片段。

## draw.io XML 基础样式

通用画布：

- `grid="0"`，避免网格显示。
- `page="1"`，设置合适页面大小。
- 所有形状使用 `strokeColor=#000000`。
- 填充使用 `fillColor=#ffffff` 或透明。
- 线条默认 `endArrow=none`；流程图、活动图、时序图按语义使用箭头。
- 中文字体：`fontFamily=Microsoft YaHei,SimSun`。
- 不使用阴影：`shadow=0`。
- 不用渐变、彩色背景、装饰图案。

## 实体-联系图规范

本 skill 的实体-联系图按用户当前确认标准执行：

1. 实体用矩形表示。
2. 属性用椭圆表示。
3. 关系用菱形表示，菱形内写关系名称。
4. 实体与关系之间用直线连接。
5. 连接线上标注基数关系，默认使用中文“一”“多”，必要时写“一对一”“一对多”“多对多”。
6. 主键属性可在椭圆文字中写“主键”，外键属性写“外键”。不要使用英文缩写，除非用户要求。
7. 如存在动态结构字段，例如大模型返回数组、非结构化报告，可画成虚线椭圆属性，并用中文说明。
8. 业务维护关系、可选关系或非强外键关系可以用虚线连接，并在图下注明含义。
9. 实体-联系图不等同于数据库表结构图。不要把表格字段列表直接堆成类图；必须体现实体、属性、菱形关系和基数。
10. 默认满足第三范式表达：实体边界清楚，属性原子化，避免把多个业务对象混在一个实体里。

实体-联系图常用关系：

- 用户 一 上传 多 皮肤图像。
- 用户 一 发起 多 诊断记录。
- 用户 一 归属 多 病历档案。
- 诊断记录 一 生成 一 病历档案。
- 管理员 一 维护 多 药品库存；如表结构无强外键，用虚线表示业务维护关系。

## 用例图规范

联网检索 UML 用例图标准后执行。默认规则：

1. 参与者用小人或 `actor` 图形。
2. 系统边界用矩形框，框名写系统名称。
3. 用例用椭圆，写动宾短语，例如“上传皮损图像”“查看诊断结果”。
4. 参与者和用例之间用无箭头实线表示通信关联。
5. `包含` 和 `扩展` 只在语义必要时使用；不要为了复杂而滥用。
6. 多角色系统要区分患者、医生、管理员等角色，不要把所有操作都连到一个“用户”。
7. 用例图表达“谁使用系统完成什么目标”，不表达先后流程。

## 流程图规范

联网检索流程图符号标准后执行。默认规则：

1. 开始、结束使用圆角矩形或终止符。
2. 处理步骤使用矩形。
3. 判断使用菱形，分支线上标注“是”“否”或具体条件。
4. 输入输出可使用平行四边形。
5. 数据存储可使用数据库圆柱。
6. 箭头方向必须清楚，尽量自上而下或自左向右。
7. 一个判断节点至少要有两个明确出口。
8. 不要把长段正文塞进节点，节点文字控制在短句。

## 活动图规范

联网检索 UML 活动图标准后执行。默认规则：

1. 初始节点用实心圆，结束节点用带外圈的终止节点。
2. 活动用圆角矩形。
3. 判断和合并用菱形。
4. 并发开始和并发汇合用粗横线或粗竖线。
5. 若涉及多个角色，使用泳道；泳道标题写角色名称。
6. 活动图表达行为流和责任边界，不要写成系统模块清单。

## 时序图规范

联网检索 UML 时序图标准后执行。默认规则：

1. 顶部放参与对象，下面是生命线。
2. 同步调用用实线箭头，返回可用虚线箭头。
3. 激活条表示对象执行控制。
4. 时间顺序从上到下。
5. 适用于接口调用、前后端交互、模型推理过程。
6. 不要把数据库表字段画进时序图。

## 功能模块图规范

功能模块图不是标准 UML 图，但论文中常用于展示系统功能分解。默认规则：

1. 顶层是系统名称。
2. 第二层是一级模块，例如用户管理、数据管理、模型评估、在线诊断、药品库存等。
3. 第三层是具体功能点。
4. 使用树状结构或分层矩形结构，不使用流程箭头表达先后顺序。
5. 模块名称使用名词短语，不使用长句。
6. 如果模块之间有数据依赖，另画架构图或数据流图，不要混在模块划分图里。

## 系统架构图规范

联网检索架构图表达习惯或项目要求后执行。默认规则：

1. 分层表达：表现层、业务逻辑层、模型服务层、数据持久层。
2. 每层用横向区域或框组表示。
3. 箭头表示调用或数据流向。
4. 数据库、缓存、文件存储等基础设施放在底层。
5. 不要把架构图画成页面截图清单。

## 数据流图规范

联网检索数据流图符号后执行。默认规则：

1. 外部实体用矩形。
2. 处理过程用圆角矩形或圆形。
3. 数据存储用开放矩形或数据库符号。
4. 数据流用带箭头连线，线上写数据名称。
5. 数据流图强调数据从哪里来、经过什么处理、到哪里去，不强调界面操作顺序。

## 中文与版式要求

1. 默认所有可见文字为中文。
2. 数据库字段名转译为中文，如：
   - `id` 写“编号”或“主键编号”。
   - `user_id` 写“用户外键”或“关联用户”。
   - `image_path` 写“图像路径”。
   - `risk_level` 写“风险等级”。
   - `medical_advice` 写“医疗建议”。
3. 基数不要写 `1:N`，默认写“一：多”或在线上分别写“一”“多”。
4. 图名、实体名、属性名、关系名不要混用英文，除非用户明确要求保留英文表名或字段名。
5. 图形排布要留白，避免线条交叉；无法避免时，优先调整布局。
6. 图中文字不要压线，不要超出图形边界。

## 交付前检查

交付 XML 前逐项检查：

1. 是否已联网检索对应图类标准。
2. 是否使用了正确图形符号。
3. 是否全中文。
4. 是否黑白、无网格、无装饰色。
5. 是否包含图号和图名。
6. 是否能被 diagrams.net 导入。
7. 是否没有把其他图类混画。
8. 实体-联系图是否有菱形关系和基数标注。
9. 流程图和活动图是否有开始、结束和清楚箭头。
10. 用例图是否有参与者、系统边界和用例椭圆。
