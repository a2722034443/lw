# 选题评估 SOP

## 触发场景

- 用户给出论文题目、研究方向、关键词或“我想做一个系统”。
- 用户不确定题目是否适合本科毕业论文。
- 用户需要把题目落成可实现的软件项目、实验和论文结构。

## 输入要求

必须尽量获取或推断：

- 题目或方向。
- 学校/学院/专业，若未知按软件项目类本科毕设处理。
- 用户可用技术栈、时间、数据来源、是否需要系统实现。
- 是否已有导师要求、模板、项目代码、数据集。

## 强制联网

正式选题评估前必须联网检索：

- `题目关键词 + 本科毕业设计`
- `题目关键词 + 系统设计与实现`
- `题目关键词 + 研究现状`
- `软件工程 本科毕业设计 需求分析 总体设计 测试 论文结构`
- 学校名称 + `本科毕业论文 格式要求` 或 `毕业设计说明书 模板`

如果无法联网，只能输出“选题预评估草稿”，不能断言研究现状、文献和创新点。

## 输出文件

在论文工作区生成：

- `topic_brief.json`
- `topic_brief.md`

内容必须包含：

- 题目规范化建议。
- 研究对象和问题边界。
- 拟实现系统名称。
- 用户角色和核心场景。
- 功能模块候选。
- 数据来源和实验/测试需求。
- 可行性判断：技术、数据、时间、论文表达。
- 风险清单和规避策略。
- 创新点或工程亮点，不得夸大。
- 推荐论文结构。
- 下一步应调用的 Skill。

## 禁止事项

- 不得把“普通管理系统”包装成不存在的重大科研创新。
- 不得承诺没有数据来源的算法实验。
- 不得写“国内外研究已非常成熟”等无来源判断。
- 不得生成虚假参考文献。

## 操作步骤

1. 明确题目核心名词：对象、方法、系统、用户。
2. 联网检索同类论文、系统和技术路线。
3. 判断题目类型：管理系统、AI/算法系统、数据分析系统、文档处理工具、移动/小程序系统等。
4. 输出系统边界：做什么、不做什么。
5. 输出论文可写性：是否能写出需求、设计、实现、测试、实验。
6. 输出风险：数据不可得、功能过大、实验难跑、创新点弱、模板不明。
7. 给出题目优化版本，通常 2-3 个。
8. 指定下一步：`literature-manager-cn` 和 `thesis-outline-cn`。

## CLI 示例

```powershell
python -m local_thesis_assistant.thesis_flow init --title "面向本科毕业论文的本地化格式检查与自动修正助手设计与实现" --type software-project --workspace local_thesis_assistant\outputs\flow_demo
```

## 交付前检查

- 题目是否能落成一个可演示的软件系统。
- 是否明确用户、功能、数据和测试。
- 是否列出不能自动完成或高风险的部分。
- 是否没有编造文献和实验结果。
- 是否给出下一步工作区文件路径。

## 衔接顺序

上游：用户题目或方向。  
下游：`literature-manager-cn` 检索文献，`project-design-cn` 设计项目，`thesis-outline-cn` 生成论文框架。
